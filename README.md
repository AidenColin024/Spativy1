Werkafspraken 
Samenwerking
 
Afwezigheid
Bij afwezigheid melden we dit bij Erik en informeren we degene die aanwezig is.
Werk dat gepland stond wordt indien nodig tijdelijk verdeeld zodat de sprintplanning zo goed mogelijk gehaald wordt.
 
Achterstand
Wanneer iemand achterloopt op planning wordt dit direct gemeld bij Erik.
Daarna bespreken we samen hoe de achterstand opgelost kan worden, bijvoorbeeld door taken opnieuw te verdelen of prioriteiten aan te passen.
 
Verdeling van taken
Taken worden eerlijk verdeeld binnen het team.
Per sprint voert iedereen ongeveer de helft van de werkzaamheden uit.
 
GIT strategie
We werken nooit in main branch alleen in feature branches en op einde mergin we eerst naar dev branch en als dan alles goed is zonder fouten naar main branchen.
 
Commits
We maken regelmatig commits zodat voortgang zichtbaar blijft.
Commit messages moeten duidelijk en inhoudelijk zijn.
 
Pull Requests
Wanneer een feature klaar is:
wordt de branch gepusht naar GitHub
maken we een Pull Request
controleren we de code voordat deze gemerged wordt
 
Conventies
Per methode een comment vooruiutleg wat de methode doet
 
workflow aanpak is het zelfde als hoe het in teams staat dus: (user stories --> use cases --> features --> taken)
